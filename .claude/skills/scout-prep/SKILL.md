# Scout Prep — 求人票からスカウト準備を一括生成

このスキルは scout-pm から呼び出されるサブスキル。単体では発火しない。

## 前提: 合同会社Plus Xについて

ヘッドハンティング会社「合同会社Plus X」（代表: 藤本）のスカウト業務で使用する。自社採用ではなく、クライアント企業のポジションを候補者に紹介する立場。

この前提から導かれる絶対ルール:
- スカウト文面に**クライアント企業名・プロダクト名を記載しない**（企業が特定できる情報は一切含めない）
- **勤務地は「東京都」レベルまで**に留める（区や住所は記載しない）
- **年収レンジとSO制度**は記載OK
- 送信者は「合同会社Plus X 代表 藤本」
- **資金調達情報は流動的に表現する**。「100億円以上の大型調達を実施済み」のように幅を持たせ、具体的な金額・時期は避ける
- **業界の表現**: 「医療系」→「ヘルスケアテック」のように、より広い印象を与え企業特定リスクを下げる表現を優先
- **すべての媒体で日本語**のテンプレートを作成する

## 過去求人の参照

prep 実行前に `data/jobs/` 配下の既存 prep.yaml を走査し、同職種・類似スキルの過去ペルソナ・検索条件を参考情報として活用する。scout-pm から過去求人情報が渡される場合はそれを使う。

## 求人票の構造化解析

求人票から以下の情報を抽出し、構造化する。求人票に明示されていない情報は無理に推測せず、書かれている範囲で最善を尽くす。

### 抽出項目

| カテゴリ | 抽出する情報 |
|---------|------------|
| **基本情報** | 企業名（内部管理用、文面には非記載）、ポジション名、雇用形態 |
| **募集背景** | なぜ今このポジションが必要なのか |
| **業務内容** | 主要な3〜5つの業務を箇条書き |
| **必須条件** | 必須スキル・経験を箇条書き |
| **歓迎条件** | 歓迎スキル・経験を箇条書き |
| **人物像** | ソフトスキル・カルチャーフィット |
| **条件面** | 年収レンジ、勤務地、リモート可否、SO制度 |
| **訴求ポイント** | 候補者に響きそうな魅力を3〜5つ。企業名を伏せた状態で伝えられる形に翻訳する |
| **技術環境** | 技術スタック・開発環境（該当する場合） |
| **選考フロー** | 面接回数・プロセス（該当する場合） |

テンプレート: `.claude/skills/scout-prep/templates/job-requirements.md` を参照。

## ターゲットペルソナの定義

「どういう人を探すのか」を具体的に描く。ここが曖昧だと検索条件もメール文面もぼやける。

### メインターゲットとサブターゲットの2パターンを作成

各ペルソナに含める要素:
- **現在の想定職種**（どんな肩書の人か。日本語と英語の両方）
- **現在の想定企業群**（具体的な企業名を5〜10社。同業界・同規模・類似課題を持つ企業）
- **経験年数の目安**
- **保有していそうなスキルセット**（ハードスキルとソフトスキルの両方）
- **転職動機として響きそうなポイント**（現職で感じていそうな課題やキャリアの次のステップ）
- **検索に使えるキーワード**（日本語・英語の両方。媒体横断で使えるベースキーワード）

### ペルソナの具体性について

悪い例: 「IT企業でPM経験のある人」
良い例: 「SaaS企業（freee、SmartHR、マネーフォワード等）で2〜3年以上のPdM経験があり、B2B向けプロダクトのロードマップ策定と開発チームとの協業を主導してきた人。30代前半で、現職では裁量の限界を感じ始めている可能性がある」

テンプレート: `.claude/skills/scout-prep/templates/persona.md` を参照。

## 候補者評価フレームワーク

候補者を見つけた後の判断基準。この求人に特化した形で作成する。

### マッチ度評価（5段階）

正本は prep.yaml の `evaluation_framework.score_criteria` に保存される。ここでは概要のみ記載する:
- **5**: 即戦力。必須+歓迎を広くカバー
- **4**: 十分マッチ。必須をほぼ満たし歓迎も一部
- **3**: ポテンシャル。必須の大半は満たすが一部不足
- **2**: キャリアチェンジ前提
- **1**: 合致しない

求人ごとの具体的な判断基準は `evaluation_framework.job_specific_criteria` に記載する。

### この求人に即した評価ポイント

求人票の必須・歓迎条件をもとに、「何を見ればスコアを判断できるか」を具体的に記載する。例:
- 「◯◯の経験年数が3年以上あるか」→ スコア4以上の条件
- 「△△業界での実務経験があるか」→ スコア+1の加点要素
- 「マネジメント経験があるか」→ 必須ではないが歓迎

### 着目ポイントの作成ルール（マッチ度4以上の候補者用）

- **3点程度**を箇条書きで作成
- 各ポイントは「◎」で始める
- 候補者の**強み・市場価値を褒める表現**を必ず入れる
- 推測に基づく部分は「**〜ではないかとお見受けしています**」と明記
- 求人の必須・歓迎スキルとの具体的な接点を示す

例:
```
◎〇〇業界にて約10年にわたりデータマネジメント領域で経験を積まれており、
  特に全社的なデータガバナンス体制の構築をリードされたご実績は非常に希少性が高いと考えております。
◎ISMSの導入・運用にも携わられており、情報セキュリティの観点からもデータの取り扱いに
  精通されているのではないかとお見受けしています。
◎直近では組織横断のプロジェクトマネジメントもご経験されており、本ポジションで求められる
  ステークホルダーとの連携力をお持ちではないかとお見受けしています。
```

## 検索条件の生成

検索条件は**媒体非依存の共通キーワード**のみ出力する。媒体固有のフィルタマッピングは scout-operator 側で行う。

出力する検索条件:
- **keywords**: メインの検索キーワード（職種・スキル中心）
- **broad_keywords**: 広め条件用のキーワード（大きめのカテゴリ）
- **narrow_keywords**: 絞り込み条件用のキーワード（具体的なスキル・技術）

テンプレート: `.claude/skills/scout-prep/templates/search-conditions.md` を参照。

## スカウトメール共通文面の生成

共通テンプレートベースの文面を生成する。媒体ごとのトーン調整は scout-operator が行う。

### 絶対に守るルール

- **クライアント企業名・プロダクト名を絶対に記載しない**。業界や事業の特徴は伝えてよいが、企業が特定できる固有名詞は含めない
- **資金調達情報は流動的に表現する**。具体的な金額・時期は企業特定につながるため、「100億円以上の大型調達を昨年実施済み」のように幅を持たせた表現にする
- **業界の表現**: 「医療系」よりも「ヘルスケアテック」のような表現を優先する
- 送信者は「合同会社Plus X 代表 藤本」
- `{候補者名}` のようなプレースホルダーを使い、パーソナライズすべき箇所を明示する
- 着目ポイント `{着目ポイント}` はプレースホルダーとして記載し、実際のリサーチ時に候補者ごとに作成する

テンプレート: `.claude/skills/scout-prep/templates/scout-message.md` を参照。

## 出力契約: prep.yaml

出力は `data/jobs/<job-id>/prep.yaml` に保存する。

### スキーマ

```yaml
schema_version: 1
job_id: <job-id>
client: <企業名>              # スカウト文には記載しない
created_at: <YYYY-MM-DD>
requirements:
  title: <ポジション名>
  must_have: [<必須スキル>, ...]
  nice_to_have: [<歓迎スキル>, ...]
  years_experience: <年数>
  salary_range: "<年収レンジ>"   # 非公開の場合は null
  notes: "<補足事項>"
personas:
  - name: <ペルソナ名>
    background: "<背景説明>"
    keywords: [<キーワード>, ...]
    target_companies: [<想定企業名>, ...]
    target_titles: [<想定職種>, ...]
    career_motivation: "<転職動機として響きそうなポイント>"
  - name: <ペルソナ名>
    background: "<背景説明>"
    keywords: [<キーワード>, ...]
    target_companies: [<想定企業名>, ...]
    target_titles: [<想定職種>, ...]
    career_motivation: "<転職動機として響きそうなポイント>"
evaluation_framework:
  score_criteria:
    5: "必須スキルをすべて満たし、歓迎スキルも複数該当。即戦力として高い確信"
    4: "必須スキルをほぼ満たし、歓迎スキルの一部も該当"
    3: "必須スキルの大半は満たすが一部不足。ポテンシャル採用の可能性"
    2: "必須スキルの一部のみ該当。キャリアチェンジ前提なら可能性あり"
    1: "求人要件とほぼ合致しない"
  job_specific_criteria:
    - "<この求人固有の判断基準。例: Go経験3年以上→スコア4以上>"
    - "<例: マネジメント経験→加点要素>"
  bonus_factors: [<加点要素>, ...]
  attention_point_rules: |
    マッチ度4以上の候補者に3点程度の着目ポイントを作成。
    各ポイントは「◎」で始め、推測は「〜ではないかとお見受けしています」と明記。
search_conditions:
  keywords: [<メインキーワード>, ...]       # 媒体非依存の共通キーワード
  broad_keywords: [<広めキーワード>, ...]
  narrow_keywords: [<絞り込みキーワード>, ...]
scout_message_base: |
  （共通テンプレートベースの文面）
```

### job-id の命名規則

`<企業名略称>-<ポジション略称>` をケバブケースで。例: `acme-backend-eng`
