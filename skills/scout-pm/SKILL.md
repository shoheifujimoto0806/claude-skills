# Scout PM — スカウト準備を統括するPMエージェント

## このスキルの役割

このスキルは「プロジェクトマネージャー」として、スカウト準備の全体を指揮する。具体的には：

1. **求人票の分析**（このファイル内で処理）
2. **ペルソナ定義**（このファイル内で処理）
3. **候補者評価フレームワーク**（このファイル内で処理）
4. **媒体別の検索条件・スカウト文面の生成**（各媒体の専門リファレンスファイルに委任）
5. **候補者プールの管理**（このファイル内で処理）

この分離により、各媒体のロジックを独立して改善できる。LinkedIn の文面を改善したいなら `references/linkedin.md` だけを修正すればよい。

## 前提：合同会社Plus Xについて

ヘッドハンティング会社「合同会社Plus X」（代表：藤本）のスカウト業務で使用する。自社採用ではなく、クライアント企業のポジションを候補者に紹介する立場。

この前提から導かれる絶対ルール：
- スカウト文面に**クライアント企業名・プロダクト名を記載しない**（企業が特定できる情報は一切含めない）
- **勤務地は「東京都」レベルまで**に留める（区や住所は記載しない）
- **年収レンジとSO制度**は記載OK
- 送信者は「合同会社Plus X 代表 藤本」
- **資金調達情報は流動的に表現する**。「100億円以上の大型調達を実施済み」のように幅を持たせ、具体的な金額・時期は避ける
- **業界の表現**：「医療系」→「ヘルスケアテック」のように、より広い印象を与え企業特定リスクを下げる表現を優先
- **すべての媒体で日本語**のテンプレートを作成する

```
  │
  ▼
[Step 3] 候補者評価フレームワーク ← このファイル
  │

[Step 4] 媒体別の検索条件＋スカウト文面生成
  │   ├── references/linkedin.md を読んで LinkedIn 向けを生成
  │   ├── references/openwork.md を読んで OpenWork 向けを生成
  │   ├── references/rds.md を読んで Recruit Direct Scout 向けを生成
  │   └── references/youtrust.md を読んで YouTrust 向けを生成
  │
  ▼
[Step 5] 候補者プール（Excel）の確認・更新
  │
  ▼
[出力] 媒体別Markdownファイル + 候補者トラッキングExcel
  │
  ▼
[Step 6（任意）] ブラウザ操作による候補者リストアップ
  │   ├── LinkedIn：プロジェクト作成 → 検索 → パイプラインに保存
  │   ├── OpenWork：検索 → ターゲットリスト作成 → 候補者追加
  │   ├── Recruit Direct Scout：スカウトルーム作成 → 検索 → リストアップ
  │   └── YouTrust：検索 → 「気になる」 → リスト作成 → 候補者追加
```

**重要：Step 4 では、必ず各媒体のリファレンスファイルを `Read` ツールで読み込んでから生成すること。** リファレンスファイルには、各媒体のUIに最適化された検索条件フォーマットや、トーン・文字数・URL有無の細かいルールが含まれている。これを読まずに生成すると品質が大幅に下がる。

## Step 1：求人票の構造化解析

求人票から以下の情報を抽出し、構造化する。求人票に明示されていない情報は無理に推測せず、求人票に書かれている範囲で最善を尽くす。

### 抽出項目

| カテゴリ | 抽出する情報 |
|---------|------------|
| **基本情報** | 企業名（内部管理用、文面には非記載）、ポジション名、雇用形態 |
| **募集背景** | なぜ今このポジションが必要なのか |
| **業務内容** | 主要な3〜5つの業務を箇条書き |
| **必須条件** | 必須スキル・経験を箇条書き |
| **歓迎条件** | 歓迎スキル・経験を箇条書き |
| **人物像** | ソフトスキル・カルチャーフィット |
| **条件面** | 年収レンジ、勤務地、リモート可否、SO制度 |
| **訴求ポイント** | 候補者に響きそうな魅力を3〜5つ。企業名を伏せた状態で伝えられる形に翻訳する |
| **技術環境** | 技術スタック・開発環境（該当する場合） |
| **選考フロー** | 面接回数・プロセス（該当する場合） |

### 職種カテゴリの判定

以下のカテゴリから、最も近いものを1つ選ぶ。候補者プールの管理に使用する。

**エンジニア系：** バックエンド / フロントエンド / フルスタック / モバイル / インフラ・SRE / データエンジニア / ML・AIエンジニア / セキュリティ / QA / CTO・VPoE・EM

**ビジネス系：** PM・PdM / 事業開発・BizDev / 営業 / マーケティング / カスタマーサクセス / 人事・HR / 経理・財務 / 法務 / 経営企画 / COO・CFO・CHRO

**デザイン系：** UIデザイナー / UXデザイナー / プロダクトデザイナー

判定した職種カテゴリは出力の冒頭に記載し、候補者プールのシート管理に使う。
## Step 2：ターゲットペルソナの定義

「どういう人を探すのか」を具体的に描く。ここが曖昧だと、検索条件もメール文面もぼやける。

### メインターゲットとサブターゲットの2パターンを作成

各ペルソナに含める要素：

- **現在の想定職種**（どんな肩書の人か。日本語と英語の両方）
- **現在の想定企業群**（具体的な企業名を5〜10社。同業界・同規模・類似課題を持つ企業。Web検索で最新情報を確認して実在する企業名を使う）
- **経験年数の目安**
- **保有していそうなスキルセット**（ハードスキルとソフトスキルの両方）
- **転職動機として響きそうなポイント**（現職で感じていそうな課題やキャリアの次のステップ）
- **検索に使えるキーワード**（日本語・英語の両方。媒体横断で使えるベースキーワード）

### ペルソナの具体性について

悪い例：「IT企業でPM経験のある人」
良い例：「SaaS企業（freee、SmartHR、マネーフォワード等）で2〜3年以上のPdM経験があり、B2B向けプロダクトのロードマップ策定と開発チームとの協業を主導してきた人。30代前半で、現職では裁量の限界を感じ始めている可能性がある」

この具体性があることで、検索条件が的確になり、スカウト文面の訴求ポイントも絞れる。

## Step 3：候補者評価フレームワーク

候補者を見つけた後の判断基準。この求人に特化した形で作成する。

### マッチ度評価（5段階）

| スコア | 基準 |
|--------|------|
| 5 | 必須スキルをすべて満たし、歓迎スキルも複数該当。即戦力として高い確信がある |
| 4 | 必須スキルをほぼ満たし、歓迎スキルの一部も該当。十分にマッチする |
| 3 | 必須スキルの大半は満たすが、一部に不足あり。ポテンシャル採用の可能性 |
| 2 | 必須スキルの一部のみ該当。キャリアチェンジ前提なら可能性あり |
| 1 | 求人要件とほぼ合致しない |

### この求人に即した評価ポイント

求人票の必須・歓迎条件をもとに、「何を見ればスコアを判断できるか」を具体的に記載する。例えば：

- 「◯◯の経験年数が3年以上あるか」→ スコア4以上の条件
- 「△△業界での実務経験があるか」→ スコア+1の加点要素
- 「マネジメント経験があるか」→ 必須ではないが歓迎

### 着目ポイントの作成ルール（マッチ度4以上の候補者用）

- **3点程度**を箇条書きで作成
- 各ポイントは「◎」で始める
- 候補者の**強み・市場価値を褒める表現**を必ず入れる
- 推測に基づく部分は「**〜ではないかとお見受けしています**」と明記
- 求人の必須・歓迎スキルとの具体的な接点を示す

例：
```
◎〇〇業界にて約10年にわたりデータマネジメント領域で経験を積まれており、
  特に全社的なデータガバナンス体制の構築をリードされたご実績は非常に希少性が高いと考えております。
◎ISMSの導入・運用にも携わられており、情報セキュリティの観点からもデータの取り扱いに精通されているのではないかとお見受けしています。
◎直近では組織横断のプロジェクトマネジメントもご経験されており、本ポジションで求められるステークホルダーとの連携力をお持ちではないかとお見受けしています。
```

## Step 4：媒体別の検索条件＋スカウト文面の生成

ここからは各媒体の専門リファレンスファイルに委任する。**必ず各ファイルを `Read` ツールで読み込んでから、そのファイルの指示に従って生成すること。**

生成する順序：

1. `references/linkedin.md` を読み、LinkedIn 向けの検索条件とスカウト文面を生成
2. `references/openwork.md` を読み、OpenWork 向けを生成
3. `references/rds.md` を読み、Recruit Direct Scout 向けを生成
4. `references/youtrust.md` を読み、YouTrust 向けを生成

各媒体の出力は、**1つの統合Markdownファイル内に媒体別セクションとして**まとめる。

特定の媒体だけを依頼された場合（例：「LinkedInのスカウト文だけ作って」）は、該当する媒体のリファレンスファイルだけを読み、その媒体分だけ出力すればよい。

## Step 5：候補者プール（職種別Excel）の管理

スカウト業務を重ねるほど「以前見た候補者」が蓄積される。これを職種カテゴリ別にExcelで管理し、新規案件のときに過去の候補者を再活用できるようにする。

### プールの仕組み

- 職種カテゴリごとに1シート（例：「バックエンド」「PM・PdM」「営業」）
- ファイル名：`candidate_pool.xlsx`
- 出力先のフォルダに既存のプールファイルがあれば更新、なければ新規作成

### プールのカラム構成

| カラム | 内容 |
|--------|------|
| 候補者名 | 氏名（リサーチ後に記入） |
| 媒体 | どの媒体で見つけたか |
| プロフィールURL | 媒体上のプロフィールURL |
| 現在の企業・職種 | 現職の情報 |
| 主要スキル | カンマ区切り |
| マッチ度 | 5段階評価 |
| 着目ポイント | 簡潔にメモ |
| スカウト送信日 | 送信した日付 |
| 返信有無 | あり/なし/未送信 |
| 案件名 | どの求人案件でスカウトしたか |
| 備考 | 自由記入 |

### 新規案件でのプール活用

求人票を受け取った際、同じ職種カテゴリの既存プールがあれば：
1. プール内の候補者を確認し、新しい案件にもマッチしそうな候補者がいないかチェック
2. 該当する候補者がいれば、再スカウトの候補としてリストアップ
3. 新たに見つけた候補者はプールに追加

## Step 6（任意）：ブラウザ操作による候補者リストアップ

Step 1〜5 で準備が完了した後、ユーザーから「実際に各媒体で候補者を探して」「リストアップもやって」と依頼があった場合、Claude in Chrome を使って各媒体のブラウザ操作を行う。

各媒体のリファレンスファイル末尾にある「ブラウザ操作ワークフロー」セクションに、媒体ごとの具体的な操作手順が記載されている。

### 媒体ごとの操作の流れ（概要）

| 媒体 | 最初にやること | 検索後のアクション | URL |
|------|------------|----------------|-----|
| **LinkedIn** | 新規プロジェクトを作成（求人名で） | 「パイプラインに保存」で候補者をリスト化 | https://www.linkedin.com/talent/home |
| **OpenWork** | 検索画面にアクセス | ターゲットリストを新規作成し候補者を追加 | https://recruiting.vorkers.com/scout/candidates |
| **Recruit Direct Scout** | **スカウトルームを先に作成**（求人名で） | ルーム内で検索しリストアップ | https://ats.hrtech.rikunabi.com/client/scouting/scoutroom/list |
| **YouTrust** | 検索画面にアクセス | 「気になる」→ 新規リスト作成（求人名で）→ リストに追加 | https://admin.youtrust.jp/headhunter/search_users |

### 注意事項
- ブラウザ操作は**ユーザーの明示的な依頼があった場合のみ**実行する（準備シートの生成だけなら Step 5 まででOK）
- 各媒体にログイン済みであることが前提。ログインしていない場合はユーザーに確認する
- 操作の詳細手順は各媒体のリファレンスファイルを参照すること
